# BUILD: Meeting-Minutes Extractor with Agent Routing (Option A)

## Why this exists

Today the meeting-minutes flow extracts action items via Claude and auto-creates
tasks. Those tasks land with no agent_type set, so the existing Research /
Outreach / Triage agents never fire on them. The user's day-to-day work is
80% generated by meetings, so this means the agent loop is effectively dormant
for the highest-volume task source.

This build closes the loop: the extractor will ALSO classify each action item
with an agent_type, the meeting-minutes route will persist that classification
when it creates the task, and the dispatcher will fire inline so each task's
agent output is ready within ~30s of the meeting being saved.

## Hard constraints

- All edits to the extractor must remain backwards-compatible: callers that ignore the new `agent_type` field on each action item must continue to work.

## What to change

### 1. src/services/actionItemExtractor.js

**Extend the SYSTEM_PROMPT** so the model returns an additional `agent_type` field per action item. Valid values:
- `"research"` — task requires web search, competitive analysis, market sizing, benchmarking, or fact-finding before it can be done. (Roughly matches /research|investigate|analyze|compare|evaluate|study|benchmark/i in spirit.)
- `"draft"` — task is to write outreach: email, proposal, presentation, follow-up, message. The user just needs the words drafted; they will review and send.
- `"none"` — task is operational/manual work that no current agent can do (e.g. "Share MVP credentials with advisor", "Pay invoice 1234", "Schedule follow-up call"). The user will do it themselves.

The prompt MUST explicitly tell Claude:
- If a task could plausibly be either research or draft, prefer "draft" only when the verb is unambiguously about producing communication ("Send X to Y", "Draft email to Z"). Otherwise pick research or none.
- Be conservative — when in doubt, pick "none". An over-classified task wastes Claude tokens; an under-classified one stays in the user's hands which is the current default anyway.
- Spanish action items count too — Claude should handle bilingual notes naturally.

**Extend the output schema** documented in the system prompt to:
```
{
  "summary": string,
  "action_items": [{
    "title": string,
    "description": string,
    "priority": "low"|"medium"|"high"|"critical",
    "assignee_hint": string,
    "due_in_days": number,
    "agent_type": "research"|"draft"|"none"
  }]
}
```

**Extend `extractActionItems()` to validate + normalize** the new field. If a returned item omits `agent_type` or returns something invalid, coerce to `"none"`. Keep the existing 12-item cap, the title slice(0,500), priority enum validation, due_in_days clamp.

**Do not introduce a new model. Do not change MODEL constant. Do not break the existing return shape — only extend it.**

### 2. src/routes/meeting-minutes.js — processMeetingAsync

Where it currently does `await Task.create({...})` for each extracted item, ALSO set:
- `agent_type`: from `item.agent_type` if `"research"` or `"draft"`, otherwise null
- `agent_status`: `"pending"` if agent_type was set, otherwise null

After the loop that creates tasks, build an array of just the IDs of tasks that have `agent_type` set, then fire a **throttled dispatch**:
- Concurrency 3. Use a simple loop with `Promise.all` over batches of 3.
- 1 second delay between batches.
- Each dispatch calls `dispatcher.processTaskById(id)` — don't reinvent.
- Wrap each call in try/catch so one failure does not stop the rest.
- Log per-task results with the existing console.log pattern (`[D2AI-MeetingMinutes] auto-dispatch ...`).

This MUST be non-blocking from the user's perspective — keep the existing `setImmediate` wrapper around `processMeetingAsync` and let it run in background. The HTTP response for saving meeting minutes goes back instantly; the user sees agent output appear ~30-60s later.
