01Executive Summary

Digit2AI is an AI-Native delivery factory — a coordinated stack of specialized AI agents, orchestrated by a single MCP brain, that converts a natural-language business problem into production software, decks, voice agents, and live operations.

The platform is a multi-tenant Express + PostgreSQL system on Render. A monolithic core host mounts a set of self-contained vertical applications and auto-mounted client builds, all sharing a common authentication, billing, and agent-orchestration substrate. The intelligence layer runs on Anthropic Claude (Opus 4.8 / Sonnet 4.6 / Haiku 4.5) via the official SDK, with provider-agnostic adapters for voice (ElevenLabs), telephony (Twilio), and specialized detection.

The architecture's organizing principle is the Model Context Protocol (MCP): every agent shares context and hands work to the next through a common contract, and every agent is wired to the customer's real systems — CRM, billing, inbox, voice channel, ops dashboards — rather than to pasted snippets. This is what turns observability ("here is a chart") into the full clinical loop: detect → diagnose → treat.

Scope. This document covers the logical and physical architecture, the AI-Native delivery methodology, data and integration design, the intake-to-production pipeline, security and compliance posture, deployment topology, and the governing architecture decision records.

02Architecture Principles

Seven principles constrain every design decision across the factory.

#PrincipleImplication
P1AI-Native by defaultScaffolding is agent-generated and iteratively verified; human effort concentrates on judgment, review, and integration — compressing delivery to weeks.
P2MCP-first integrationAgents and connectors share context through the open Model Context Protocol; no agent is wired to a single vendor model or a single data silo.
P3Multi-tenant isolationEvery table carries tenant_id; every query is tenant-scoped. One platform serves many customers without data bleed.
P4Bilingual-nativeEN + ES are first-class across UI, voice, agents, and generated artifacts — not an afterthought translation pass.
P5Self-contained verticalsEach vertical is an independent Express Router with its own Sequelize instance and migrations — isolation without microservice operational overhead.
P6Provider-agnostic adaptersDetection, TTS, search, and model routing sit behind contracts; swap a provider via env, not a rewrite.
P7Ship-to-verifyEvery increment is deployed and health-checked; the build loop auto-fixes against live signals rather than assumptions.

03AI-Native Delivery Methodology

The factory runs a continuous five-phase loop. It is the operational backbone of every engagement, executed by the MCP brain routing across the agent workforce.

PHASE 1
Analyze
Parse NL · design system · model schema · score fit
PHASE 2
Develop
Prompts · code · Sequelize models · migrations
PHASE 3
Test
Local · API smoke · SIT · migration check
PHASE 4
Deploy
Commit · push main · Render auto-deploy
PHASE 5
Review
Health · logs · Findings · auto-fix loop

Phase 5 closes back into Phase 1: any health failure, missing owner, or unmet acceptance criterion re-enters the loop as a new task. The loop self-terminates when all components are live, green, and verified, or escalates a single explicit blocker (e.g., a missing external credential) without halting the rest of the build.

Engagement modes

  • Build Service — NL → production web app on the customer's domain.
  • BA-on-Demand — decks, plans, research, regulatory write-ups; no software.
  • Vertical Platform — deploy an existing platform to a new tenant; months → weeks.
  • Voice AI — a 24/7 inbound voice agent wired to the customer's CRM + calendar.

Token cost is estimated and recorded per call for FinOps.

A single host process composes four component classes. The host owns shared concerns (auth, billing, sessions, static serving, agent substrate); everything else mounts onto it.

Mount contract. A vertical or client build is a black box exposing an Express Router. The host mounts it at /<slug>, injects nothing but the request, and the module manages its own DB, models, and tables (created on boot via sync({alter:false})). This keeps blast radius per app and lets a new vertical ship without touching the core.

07Data Architecture

PostgreSQL on Render, accessed through Sequelize with SSL. Two logical databases separate concerns: CRM_DATABASE_URL (CRM + tenant/project data) and DATABASE_URL (legacy/secondary). Bridge endpoints that cross databases instantiate a dedicated Sequelize connection against the correct URL.

Tenancy & naming

  • Isolation: every row carries tenant_id; queries are tenant-scoped at the route layer.
  • Prefixes: each vertical namespaces its tables — df_* (Veritas), am_* (AgroMercado), d2_* (Projects), tunjo_*, etc.
  • JSONB + GIN for flexible metadata (product metadata, triage_structured) with indexed search.

Migration strategy

Models sync with { alter:false } — schema is never auto-altered in production. New columns/tables ship as explicit raw-SQL migrations (canonical .sql files per module) run at startup or manually. This trades convenience for safety: no surprise destructive ALTERs against live tenant data.

Blob & document storage

Binary assets (documents, audio, backups) go to AWS S3 with presigned URLs; generated documents use pdfkit / pdfmake / docx / html-pdf-node + headless Chromium (@sparticuz/chromium + puppeteer-core).

Architect command surface

The Architect & Builder agent exposes a deterministic command set that pipelines the funnel: triage → teaser → cost → quote → proposal → loop (autonomous end-to-end build), plus voice (Lina orb) and a2p (SHAKEN/STIR registration). Each is invokable alone; proposal silently runs missing upstream stages.

Transcript-direct intake. A dedicated client build captures spoken project descriptions client-side (Web Speech API), persists the transcript JWT-guarded, and forwards into the project inbox with triage_bypass=true so downstream knows it skipped automated triage.

10Voice AI Architecture

Two distinct voice paths share the Rachel/Ana/Lina persona layer.

Inbound telephony

Twilio number → webhook → voice route. STT + LLM + TTS pipeline qualifies the caller, books into the calendar (client-15 carve-out books d2_calendar_events), answers FAQs, hands off to a human, and logs the call + transcript to the CRM.

Web conversational orb

ElevenLabs convai with a dedicated agent ID per product (never shared). The browser SDK connects directly; no API key reaches the client (agent IDs are public by design). A client-tool bridge maps the agent's run_partnership_triage call to a browser handler that POSTs the backend.

Teaser narration uses a $0, no-key path: server-proxied Microsoft Edge "Read Aloud" neural TTS with automatic browser-speech fallback — the canonical "Lina" pattern, stampable into any page via the voice command.

Regulatory coverage

Compliance, Regulatory Researcher, and Privacy Officer specialists scope each regulated build: GDPR / HIPAA (US health), Colombian Habeas Data (Ley 1581/2012) + SFC (fintech, e.g. PLANEA), SARLAFT/AML, and sector frameworks. The AI never auto-commits the firm to a regulated engagement — every triage is a human-reviewed first read.

12Deployment & Infrastructure

  • Platform: Render. Trigger: git push origin main → auto-deploy in ~90–100s.
  • Build: build.sh; static public/ assets need no build step; React/Vite dashboards are pre-built.
  • Config: 12-factor env vars; feature flags gate optional waves (e.g. INTUITIVE_ENGAGEMENT_GO, VERITAS_*, AGROMERCADO_*).
  • Database: managed PostgreSQL (HA + backups); SSL required; connection pooling via Sequelize.
  • Ephemerality note: on-disk dirs (e.g. proposal-audio) are wiped on redeploy — durable artifacts live in S3 or the DB.
Operational caution. Any code push triggers a full redeploy; time pushes around live demos and expensive regen steps (e.g. ElevenLabs narration) accordingly.

13Observability & Site Reliability

  • Health/Debug: every app exposes /<app>/health and a /debug/<app>-error endpoint surfacing mount/runtime errors.
  • Logging: winston (structured) + morgan (HTTP); PII-scrubbed.
  • Schedulers: node-cron pollers (meeting/RSVP reminders, inbox digest, public-source refresh) — gated by the autosend flag.
  • Active monitoring: the Neural Findings agent watches every project for stalls, missing owners, and overdue milestones and pings before slippage.
  • Target: 99.9% SLA on delivered software, with ongoing AI agent support.

16Quality Attributes (NFRs)

99.9% SLA; HA Postgres; health/debug per app; auto-fix loop on live signals.
